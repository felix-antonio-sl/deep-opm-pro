opforja.Control de flujo · OR/XOR

Cuándo un proceso ocurre, se omite o elige entre ramas. El control existe para explicar la ejecución — no para decorar.

§6.3 · operadores lógicos

AND · XOR · OR

  • AND — sin arco: enlaces separados que no se tocan → todos a la vez (default).
  • XOR — un arco discontinuo sobre el abanico → exactamente uno.
  • OR — dos arcos concéntricos → al menos uno.

§6.1 · marcas de control

c · e · ¬

  • c — condición (el enlace es precondición).
  • e — evento (dispara el proceso).
  • ¬ — negación.

Badge junto al extremo del proceso. Anotan transformadores/habilitadores, no estructural ni invocación.

§6.2 · excepciones temporales

Sobretiempo y subtiempo

  • / sobretiempo — duración real > máxima declarada.
  • // subtiempo — duración real < mínima declarada.

Y la probabilidad de rama se anota Pr = p en el abanico.

§8 · familia aparte

Invocación

Un proceso llama a otro (rayo en zigzag). Es familia autónoma: no lleva c/e (AP-10). Para omisión condicional, usa un nodo de decisión o un fan de invocación.

Errores de control

Dónde NO va

  • AP-01/02 c/e sobre un resultado → muévelo al lado de entrada.
  • AP-09 c/e sobre estructural (es invariante temporal).
  • AP-28 c y e a la vez sobre el mismo enlace → control externo explícito.
  • AP-03 abanico XOR/OR de resultado con c/e.

La regla de oro

Control = pregunta de ejecución

Usa control solo cuando responde: ¿cuándo este proceso ocurre, se omite, o llama a otro? Si no responde una pregunta de ejecución, no lo pongas — es ruido.
