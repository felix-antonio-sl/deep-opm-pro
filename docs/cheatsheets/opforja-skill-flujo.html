opforja ↔ skill

Flujo de trabajo

Flujo interno skill–agente–mesa: cómo se serializa, valida y transporta el modelo. Para el diálogo entre operador y skill, usa la hoja «Hablar con la skill».

Humano
Trae el dominio y el gusto. Decide significado, ratifica. La skill no inventa contenido de dominio.
Skill · modelamiento-opm
El método Forja y la postura dialéctica. Anti-barro: bloquea el avance ante ambigüedad, exige aclarar antes de plasmar.
Mesa · opforja
El lienzo, la validación tripartita y la persistencia. Realiza; no es fuente de verdad del método.

El bucle

Loop de modelado del agente

  1. Triaje — leer el intent y el contexto del operador.
  2. Elicitar / construir — aplicar Forja; ante barro, parar y preguntar.
  3. Serializar — proto OPL-ES estricto → compilador. No fabricar JSON a mano.
  4. Validar el loop — round-trip, contención, canon, golden-harness.
  5. Revisar visual — render read-through cuando el OPD importa.

Las herramientas del agente son read-through: no mutan el proto ni el dominio.

El puente directo

mesa pull · push

La skill lee y escribe la mesa directo, sin copiar y pegar. Desde la terminal del host:

cd app && bun run mesa pull <modelo>
  # contexto del modelo para la skill

cd app && bun run mesa push <modelo> bundle.json --nota "…"
  # nueva revisión versionada

cd app && bun run mesa modelos
  # listar los del tenant

En la mesa, el chip «Revisión nueva» avisa cuando el agente empujó — se trae sin pisar tu trabajo local.

Disciplina del puente

Reglas del push

  • Pull antes de push — trabaja siempre sobre base fresca.
  • Nunca push sin validación local verde (contrato de import).
  • Respeta el 409: la mesa avanzó — re-pull, jamás forzar.
  • Base no ratificada (autosave) → pide confirmación antes de cerrar el loop.
  • Nota de versión con procedencia.

La fuente de verdad

La procedencia decide

La doctrina read-through se aplica al modelo con sello, nacido de un proto OPL-ES: el proto es la fuente y las correcciones se re-elicitan allí.

En un modelo nacido en la mesa (table-origin), el push libre es legítimo. Este carril técnico no sustituye la autoridad humana sobre el significado del dominio.

Trazabilidad

Sello de procedencia

El bundle es citable: lleva un sello de tres componentes

{ protoHash, autoriaVersion, layoutVersion }

que permite rehidratar y auditar el modelo, y detectar si la mesa lo editó tras la emisión. Formato deep-opm-pro.modelo.v0.

El diálogo

Ratificar y notas de mesa

  • Notas de mesa — preguntas ancladas a un componente; insumo de la re-elicitación, desechables.
  • Anclas [RATIFICAR] — pasan a vigentes solo por re-elicitación de la skill, con fuente.
  • El agente propone estandarizar citas/normas; no emite anclas sin confirmación humana.

Cuando no hay host

Fallback: puente W6.0

Sin acceso a la terminal, el bucle viaja por el portapapeles: Ctrl+K › «Copiar contexto para la skill» compone en un markdown la procedencia + pendientes [RATIFICAR] + notas + diagnóstico + OPL.

Se pega en la sesión de la skill, que re-elicita y re-emite el bundle. Mismo bucle, transporte manual.

Empezar sin ceremonia

Bottom-up con apuntes

No hace falta partir del SD raíz. Bosqueja fragmentos como apuntes (rigor relajado) y llévalos al modelo madre por composición, o al SD0 por el verbo «adoptar» del Taller.

El rigor se cobra al graduar, no al explorar.

Serializar y verificar

Del proto al bundle

  • Compilador de autoría — proto OPL-ES → Modelo (determinista, verificador total).
  • render:headless — PNG (ver) + SVG (diff) por OPD, pasada visual read-through.
  • verify:reproducible — golden-harness: la emisión es reproducible byte-estable.

Honestidad

Límites de la mesa

La skill conduciendo opforja no hace:

  • out-zoom automático (el in-zoom se declara, el out-zoom es manual).
  • diff de modelos · export a PDF real (la salida canónica es Markdown).
  • cosimulación federada · multiusuario concurrente por modelo.
  • razonamiento / inferencia automática (modela y valida, no infiere).
