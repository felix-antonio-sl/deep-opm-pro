# Roadmap — dirección vigente de opforja

**Fecha:** 2026-07-18

**Horizonte:** próximos cortes con evidencia o dependencia activa después del
cierre de integridad del puente mesa↔skill

Este roadmap contiene solo trabajo futuro. El estado verificado vive en
`../handoff-2026-07-18.md`; los contratos permanentes, en `../../CLAUDE.md`,
`../canon-opm/`, `../../ui-forja/` y las especificaciones citadas.

## Orden recomendado

No hay un corte activo priorizado al cierre del 2026-07-18. Los seis cortes que
originaron esta versión quedaron ejecutados, verificados y retirados de la cola;
su evidencia vive en el [handoff](../handoff-2026-07-18.md), el
[registro de conformidad](registro-conformidad-ssot.md), el
[histórico de bugs](../bugs/HISTORY.md) y Git.

La siguiente intervención se abre solo por una demanda observada, un fallo
reproducible o una decisión del operador. Hasta entonces, la acción recomendada
es vigilar producción con `cd app && bun run cordon:estado` y mantener el ledger
de bugs sin entradas resueltas bajo activos.

## Brechas declaradas, no calendarizadas

- Láminas didácticas OPD para el manual OPM puro: generarlas desde un proto
  canónico mediante el renderer headless, con fuente reproducible.
- Auth/tenants v2: invitaciones, roles efectivos y administración multiusuario;
  abrir solo ante demanda nombrada. El enlace de cada request a la identidad de
  sesión observada ya pertenece a v1 y no se reabre aquí.
- Coedición simultánea del mismo modelo: el CAS evita sobrescritura silenciosa,
  pero no fusiona intenciones ni convierte la mesa en un editor colaborativo.
- Out-zoom: capacidad ausente y declarada en
  `registro-conformidad-ssot.md`; no hay modelo productivo que la exija.
- Simulación probabilística sin pesos: requiere variante de modelo,
  serialización, interfaz, OPL y diagnóstico; esperar caso real.
- Anclaje: verbo formal `promover-a-Pieza`, resolución externa completa,
  herencia mutacional y OPL/render del metacontrato; mantener como evolución.
- GAPs visuales de `spec-forja-opd-es` §22: abrir por evidencia de usuario o
  incumplimiento de gate, no por inventario.

## Decisiones cerradas que no vuelven al backlog

- No ejecutar el macro-refactor especulativo. L7–L8 sellan las dependencias
  corregidas; un nuevo movimiento exige dolor medido o una feature bloqueada.
- En la fuente actual, la persistencia no usa «última escritura gana» para
  modelos o workspace: cada mutación se liga a una base observada; un conflicto
  se rechaza y se reconcilia. Producción lo adopta solo al cerrar el corte 1.
- El token de agente conserva mínimo privilegio: lectura y commit atómico de
  revisiones; no se rehabilitan las escrituras heredadas.
- S-2, I-3, I-4, I-6 y P-2 están cerrados: Configuración única para renombrar,
  guardas explicables, comandos humanizados, continuidad de búsqueda y borrado
  con impacto + deshacer.
- La graduación tiene una sola ruta validada. No reintroducir un toggle directo
  desde `PanelCarpetas`.
- La carrera asíncrona de Drift está cerrada con generación y prueba de
  resoluciones invertidas.
- Auth/identidad v1, mobile de solo lectura, puente mesa↔skill, Taller, vitrina,
  modo apunte, gesto de anclar, C4, Inspector, chrome de gestión y frontera
  `src/app` están cerrados.
- No convertir observaciones estéticas sin evidencia en backlog.

## Gates por corte

- Cambio documental: enlaces locales activos, punteros raíz exactos, fechas ISO
  y una sola versión visible por especie.
- Cambio de persistencia: unitarias, integración PostgreSQL aislada y pruebas
  de conflicto/rollback; nunca validar mutaciones contra producción.
- Cambio de código: `cd app && bun run check` como mínimo.
- Cambio visual: añadir `bun run lint`, `bun run build`,
  `bun run design:governance` y los E2E afectados.
- Refactor transversal o cierre: `bun run gate:refactor`, con el servidor de
  desarrollo apagado antes del smoke.
- Cambio doctrinal: flujo del custodio en `kora-pneuma`, re-pin de puentes y
  `bun run cordon:skill`.

## Criterios de selección

1. Una brecha abre corte solo con usuario, decisión o gate afectado.
2. Cada corte declara beneficiario, decisión soportada y prueba falsable.
3. El resultado se registra en un handoff nuevo; el roadmap no es bitácora.
4. Un sucesor se publica como `roadmap-AAAA-MM-DD.md` y desplaza esta versión a
   `_archivo/`.
